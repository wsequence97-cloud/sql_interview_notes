# 牛客 SQL 大厂真题导入设计

## 目标

从牛客网公开可访问的“SQL 大厂笔试真题”中，筛选与当前 10 个 SQL 题型目录匹配、尚未收录且难度为中等、较难或困难的题目，并按项目规范逐题生成 Markdown 笔记。

## 范围

- 仅处理牛客标注为中等、较难、困难的 SQL 大厂真题。
- 跳过入门、简单题目。
- 跳过与现有题目在查询目标和核心解法上实质重复的题目。
- 跳过无法从公开页面完整核验题目描述、表结构、示例数据或期望结果的题目。
- 跳过需要登录、会员或其他权限才能取得关键内容的题目，不通过推测补齐。

## 来源规则

1. 题目描述、字段、示例数据和期望结果以牛客官方题目页的公开内容为准。
2. MySQL 解法必须有牛客公开题解或公开答案作为依据，再改写成 Hive SQL。
3. 标准答案只保留改写后的 Hive CTE 完整版，不大段复制公开题解文字。
4. 每道题在“题目描述”中记录牛客题号、难度和来源链接，便于复核。
5. 来源信息不完整或互相矛盾时，记录到检查报告并跳过该题。

## 筛选与去重

先建立现有题库标题、查询目标、核心考点和所在目录的基线，再与候选题逐题比较。

去重不只比较标题，还比较：

- 输入表和业务对象是否只是换名；
- 最终查询目标是否相同；
- 核心 SQL 技法和关键口径是否相同；
- 是否能为现有题库带来新的边界条件或解法价值。

仅标题相似但核心考点不同的题目可以录入；标题不同但本质完全相同的题目跳过。

## 归类与命名

- 严格使用 `SQL题目生成格式规范.md` 中已有的 10 个目录。
- 同时涉及多个题型时，放入最核心的题型目录。
- 新文件编号使用该目录当前最大三位序号加一。
- 文件名采用简短中文题目名称，不改动已有文件路径和编号。

## 单题内容

每道新题只包含以下六部分：

1. 题目描述
2. 期望结果
3. 标准答案
4. 我的解答
5. 我的解答正确版本
6. 我的复盘要点

“标准答案”从 `with data as (...)` 开始，包含公开示例数据 CTE、解题 CTE 和最终查询，并统一使用 Hive 语法。

由于这些题目不是用户亲自提交的解答：

- “我的解答”写“本题暂未提供我的解答。”
- “我的解答正确版本”写“本题暂未提供我的解答，因此没有对应的‘我的解答正确版本’。”
- “我的复盘要点”写“本题暂未补充个人复盘要点，后续复盘时更新。”

不得替用户编造原始 SQL、卡点或个人感悟。

## Hive 改写规则

- 使用 CTE 分层组织完整 SQL。
- 日期转换优先使用 `to_date()`。
- 日期加减使用 `date_add()`、`date_sub()` 和 `datediff()`，不保留 MySQL `interval n day`。
- 将 MySQL 专有的 `if()`、`date_format()`、字符串和时间函数按 Hive 语义改写。
- 累计窗口显式声明 `rows between unbounded preceding and current row`。
- 对 `date`、`type`、`rank` 等冲突字段使用反引号或在基础 CTE 中改名。
- 最终查询补充稳定排序，但不改变题目要求的输出口径。

## 索引与报告

新增题目后同步更新：

- 对应目录的 `topic_summary.md`；
- 根目录 `README.md` 的题目数量和更新记录；
- `generation_check_report.md` 的新增文件、结构和缺少用户解答情况。

检查报告还应列出因以下原因未录入的候选题：难度过低、已重复、公开内容不足、题型不匹配。

## 验证

完成后执行以下检查：

1. 每个新增文件恰好包含六个规定章节。
2. 文件编号连续且未覆盖已有文件。
3. 每道题均保留可访问的牛客来源链接和难度。
4. 标准答案包含样例数据 CTE、解题 CTE 和最终查询。
5. 静态搜索 MySQL 专有语法，逐项确认已改为 Hive。
6. 对示例数据人工推演或使用兼容 SQL 环境核对期望结果。
7. README、题型总结和检查报告中的数量一致。

## 执行顺序

按题型目录分批处理候选题。每一批依次完成来源核验、去重、Hive 改写、单题生成和结构检查，再处理下一批；最终统一更新索引和报告。
