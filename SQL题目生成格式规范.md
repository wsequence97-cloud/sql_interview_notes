# SQL 题目生成格式规范

本文件用于约束 Codex 后续为每一道 SQL 面试题生成独立 Markdown 笔记。

当前项目是我的 SQL 刷题笔记库。后续我会给出一道 SQL 题目，可能包含：

1. 原题描述
2. 表结构
3. 示例数据
4. 期望结果
5. 标准答案
6. 我自己写的解答
7. 我与 Codex 讨论后形成的卡点、错误、思考或复盘要点

Codex 的任务是：根据我给出的题目内容，把题目归类到对应题型目录下，并生成一个专属该题目的 Markdown 文件。

---

## 1. 文件放置规则

项目中已有 10 个题型目录，请不要随意重命名目录。

| 目录                       | 题型                                          |
| ------------------------ | ------------------------------------------- |
| `01_连续登录`                | 连续登录、连续出现、连续日期类问题                           |
| `02_lead_lag_first_last` | `lead`、`lag`、`first_value`、`last_value` 类问题 |
| `03_排序窗口函数`              | `row_number`、`rank`、`dense_rank`、TopN 类问题   |
| `04_累计汇总`                | 累计和、滚动窗口、移动平均类问题                            |
| `05_炸裂函数`                | 数组、字符串拆分、侧视图、展开类问题                          |
| `06_关联应用`                | 多表关联、反关联、半连接类问题                             |
| `07_留存计算`                | 次日留存、N 日留存、用户回访类问题                          |
| `08_数据展开与收缩`             | 行列转换、宽表窄表、数据展开收缩类问题                         |
| `09_合并区间`                | 时间区间合并、重叠区间、连续区间类问题                         |
| `10_状态标记与填补缺失值`          | 状态变更、缺失值填补、分组标记类问题                          |

如果题目同时属于多个题型，优先放到最核心的题型目录中。

例如：

* 连续登录 + `row_number`，优先放入 `01_连续登录`
* 次日留存 + `join`，优先放入 `07_留存计算`
* TopN + `rank`，优先放入 `03_排序窗口函数`
* `explode` + 行转多行，优先放入 `05_炸裂函数`
* 扫描线 + 累计在线人数，优先根据业务放入 `09_合并区间` 或 `10_状态标记与填补缺失值`

如果无法判断题型，不要乱放。请先在回复中说明判断不确定，并给出最可能的 1 到 2 个目录。

---

## 2. 文件命名规则

题目文件命名格式：

```text
三位序号_题目简称.md
```

示例：

```text
001_查询连续登录3天以上的用户.md
002_查询每个用户最大连续登录天数.md
003_查询每个玩家最长连胜场次.md
```

规则：

1. 如果是新题，自动使用该目录下当前最大序号 + 1。
2. 如果是修改旧题，保持原文件名不变。
3. 文件名使用中文即可。
4. 文件名不要太长。
5. 文件名不要包含空格。
6. 不要随意改动已有题目的编号。

---

## 3. 单题 Markdown 固定结构

每道题只保留以下 6 个部分，不要额外生成其他章节。

```markdown
# 题目名称

## 1. 题目描述

## 2. 期望结果

## 3. 标准答案

## 4. 我的解答

## 5. 我的解答正确版本

## 6. 我的复盘要点
```

禁止额外生成以下内容：

* 核心步骤解释
* 记忆点
* 面试追问
* 面试口述版
* 题型定义
* 关联题型
* 可迁移模板
* 复杂长篇分析
* 多种答案版本堆叠

本项目只需要简洁、实用、贴合我个人刷题复盘的题目笔记。

---

## 4. 各部分写法要求

### 1. 题目描述

这一节用于完整记录原题。

必须包含：

1. 题目背景。
2. 表名。
3. 字段名。
4. 字段含义。
5. 示例数据。
6. 查询目标。
7. 题目中的特殊条件或口径。

如果用户给出的题目内容比较零散，需要整理成清晰的题目描述。

如果用户没有给字段类型，不要强行补复杂类型，只写字段含义即可。

推荐格式：

```markdown
给定表 `data`，字段如下：

| 字段名 | 含义 |
|---|---|
| `字段1` | 含义 |
| `字段2` | 含义 |

示例数据：

| 字段1 | 字段2 |
|---|---|
| 示例值 | 示例值 |

要求查询：……
```

如果题目有歧义，必须在题目描述末尾补充：

```markdown
> 本题口径：……
```

例如：

```markdown
> 本题口径：“连续登录 3 天以上”按 `> 3` 理解，不包含刚好 3 天。
```

---

### 2. 期望结果

这一节只放最终输出结果。

用 Markdown 表格展示。

如果用户没有给期望结果，但可以根据题目和样例数据推导，请直接推导。

如果无法推导，写：

```markdown
用户未提供期望结果，且当前题目信息不足，暂无法唯一推导。
```

不要编造不确定的期望结果。

---

### 3. 标准答案

这一节写完整可运行的标准答案。

默认使用 Hive SQL。

标准答案必须包含：

1. 样例数据 CTE。
2. 解题过程 CTE。
3. 最终查询 SQL。

也就是说，标准答案必须从 `with data as (...)` 开始，保证我复制整段 SQL 就可以直接运行。

标准答案不要只写核心 select。

标准答案默认只保留一个最推荐版本，不要堆叠多个版本。

推荐写法：

```sql
with data as (
    select stack(
        3,
        1, '2024-01-01', 'Win',
        1, '2024-01-02', 'Lose',
        1, '2024-01-03', 'Win'
    ) as (player_id, match_day, result)
),
base as (
    select
        player_id,
        to_date(match_day) as match_day,
        result
    from data
)
select
    ...
from base
;
```

Hive SQL 注意：

1. 日期转换优先使用 `to_date()`。
2. 日期加减使用 `date_add(date_col, n)` 或 `date_sub(date_col, n)`。
3. 不要使用 MySQL 的 `interval n day` 写法。
4. 窗口函数中涉及累计时，尽量显式写：

   ```sql
   rows between unbounded preceding and current row
   ```
5. 如果字段名是 `date`、`type`、`rank` 等可能冲突的词，SQL 中用反引号，或者在 CTE 中改成清晰别名。
6. 如果需要去重，要在 CTE 中明确写出。
7. 如果需要排序保证结果稳定，可以在最终查询中写 `order by`。

---

### 4. 我的解答

这一节用于原样保留我自己写的 SQL 或思路。

如果我提供了自己的 SQL，必须原样粘贴，不要擅自修改。

格式：

```sql
-- 我的原始解答
```

如果我只提供了思路，没有 SQL，就用文字原样记录。

如果我没有提供任何解答，写：

```markdown
本题暂未提供我的解答。
```

不要在这一节分析对错。

---

### 5. 我的解答正确版本

这一节是基于“我的解答”改出来的正确版本。

这一节重点关注两个问题：

1. 我的 SQL 在语法上能不能正确运行。
2. 我的解题思路是否正确、是否有更合适或更高效的写法。

不要把以下内容当成主要分析重点：

* 别名不够规范
* 字段命名不够好看
* SQL 排版不够标准
* 是否加了 `as`
* CTE 名称是否足够优雅
* 输出列名是否更像标准答案

这些属于标准答案天然应该整理好的格式问题，不要在复盘里反复强调。只有当别名、字段名、CTE 名称会导致 SQL 报错、字段歧义、后续无法引用时，才需要指出。

本节修正优先级如下：

1. 语法错误
   例如字段拼写错误、函数写法错误、Hive/MySQL 语法混用、窗口函数语法错误。

2. 逻辑错误
   例如分组字段缺失、过滤顺序错误、窗口函数使用位置错误、`where` 和 `having` 混用、`left join` 被写成 `inner join` 效果。

3. 口径错误
   例如没有去重、分母分子定义错误、连续天数是否包含当天、是否要求所有课程都有成绩、是否需要统计跨天记录。

4. 思路优化
   例如可以先过滤再关联、先聚合再 join、减少无意义的中间表、避免先做大宽表再筛选。

5. 格式规范
   仅在不影响重点的情况下顺手修正，不作为复盘重点展开。

要求：

1. 尽量保留我的原始写法和思路。
2. 不要直接换成完全不同的标准答案。
3. 如果我的思路基本正确，只修正必要错误。
4. 如果我的思路能跑但不够优，给出更优版本，并保持解释简短。
5. 如果我的思路本身不可行，再说明需要改成标准答案思路。
6. 这一节必须使用 Hive SQL。
7. 这一节可以不重复完整样例数据 CTE，除非我的原解答本身包含 CTE。
8. 这一节的重点是让我看到：我的代码应该怎么改才对，以及为什么这样更合理。

格式：

```sql
-- 基于我的解答修正后的正确版本
```

如果我没有提供我的解答，写：

```markdown
本题暂未提供我的解答，因此没有对应的“我的解答正确版本”。
```

---

### 6. 我的复盘要点

这一节是整篇笔记中最重要的个人沉淀部分。

它不是通用知识点总结，而是要记录：

1. 这道题我原来不会的地方。
2. 我写错、想错、漏想的地方。
3. 我通过和 Codex 讨论后真正理解的点。
4. 下次遇到同类题时，我应该先想到什么。
5. 这题最容易让我踩坑的具体位置。

#### 写法要求

只写 3 到 5 条，不要超过 6 条。

每条尽量控制在 1 到 2 句话内。

每条必须具体到本题，不要写空泛的话。

不要记录纯格式问题，例如别名是否规范、是否加 `as`、SQL 排版是否更好看。除非这些问题会导致 SQL 报错或逻辑歧义，否则不要写入复盘要点。

禁止写这种空泛表达：

* “注意 SQL 执行顺序”
* “注意窗口函数使用”
* “注意 group by”
* “注意字段名”
* “注意去重”
* “注意 join”

除非后面明确写出：本题中具体是哪一层、哪个字段、哪个条件容易错。

#### 推荐标签

优先使用以下标签：

```markdown
- 【我的卡点】……
- 【错误原因】……
- 【修正记法】……
- 【下次先想】……
- 【本题坑点】……
```

不要求每题都五个标签都有，只保留和本题最相关的 3 到 5 条。

#### 好的例子

```markdown
- 【我的卡点】我一开始只想到 `group by sid having min(score) > 60`，但没意识到这只能说明“已有成绩都及格”，不代表这个学生所有课程都有成绩。
- 【错误原因】如果题目要求“三门课都参加且都及格”，就必须额外判断 `count(distinct cid) = 课程总数`，否则缺考学生可能被误判为合格。
- 【下次先想】遇到“全部满足”类题目，要先确认口径：是“已有记录全部满足”，还是“应有项目全部满足”。
```

```markdown
- 【我的卡点】我知道登录打 `+1`、登出打 `-1`，但一开始没看出来这是在算“直播间峰值在线人数”。
- 【修正记法】这类题本质是扫描线：事件打点 -> 按时间累计 -> 取累计最大值。
- 【本题坑点】同一时间有人进入、有人离开时，要明确先处理 `+1` 还是 `-1`，否则峰值在线人数可能差 1。
```

```markdown
- 【我的卡点】我原来把这题当成普通 `join` 题，但它真正的关键是先确定分母用户群。
- 【错误原因】留存题如果直接 `inner join` 次日行为，会把没回来的人过滤掉，导致分母变小、留存率偏高。
- 【下次先想】留存题先问三件事：分母是谁、分子是谁、N 天后的日期条件怎么关联。
```

#### 差的例子

不要写：

```markdown
- 注意窗口函数。
- 注意 SQL 执行顺序。
- 注意 group by。
- 注意字段别名。
- 这题要认真理解。
```

这些内容不利于复盘。

如果当前还没有经过讨论，没有沉淀出个人要点，则写：

```markdown
本题暂未补充个人复盘要点，后续复盘时更新。
```

## 5. 新增题目处理流程

当我给出一道新题时，你必须按以下流程执行：

1. 阅读题目内容。
2. 判断题目所属目录。
3. 确认该目录下当前最大题号。
4. 新建对应编号的题目 md 文件。
5. 按固定 6 个部分生成内容。
6. 保留我的原始解答。
7. 写出标准答案。
8. 基于我的原始解答生成“我的解答正确版本”。
9. 把我和你讨论过的关键点整理到“我的复盘要点”。
10. 输出本次新增文件路径。

---

## 6. 修改旧题处理流程

当我要求你修改已有题目文件时，你必须：

1. 保持原文件路径不变。
2. 保持原题目名称不变，除非明显错误。
3. 删除旧版多余章节。
4. 只保留新版 6 个部分：

   * 题目描述
   * 期望结果
   * 标准答案
   * 我的解答
   * 我的解答正确版本
   * 我的复盘要点
5. 迁移旧文件中的题目描述、示例数据、期望结果、正确答案、我的原始思路。
6. 将旧版“正确答案”整理到新版“标准答案”。
7. 将旧版“我的原始思路 / 我的解答”整理到新版“我的解答”。
8. 基于旧版分析和我的原始 SQL，生成“我的解答正确版本”。
9. 将旧版“记忆点”“核心步骤解释”“注意点”中真正有价值的内容，压缩整理到“我的复盘要点”。
10. 不要保留旧版冗长讲义式解释。
11. 不要把别名、排版、是否加 `as` 这类格式问题写成复盘重点。
12. 如果旧文件里没有我的解答，不要编造，写“本题暂未提供我的解答”。

---

## 7. 输出风格要求

1. 语言使用中文。
2. 内容要简洁。
3. 不要生成长篇教学解释。
4. 不要生成无关扩展。
5. 不要堆叠多个 SQL 版本。
6. 不要替我编造题目来源。
7. 不要替我编造我的解答。
8. 不要替我编造我没有讨论过的个人思考。
9. 如果信息不足，明确写“未提供”或“暂无法确定”。
10. 优先保证 SQL 正确、结构清晰、方便我后续复盘。
11. 复盘要点要像我自己的错题本，不要像教材总结。

---

````markdown
    select
        字段1,
        字段2

    ...

```

```sql

```sql

```
````
