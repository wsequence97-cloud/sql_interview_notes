# 三、本项目的核心整理原则

非常重要：

1. 本项目不是为了收集 LeetCode / 牛客原题，而是整理我自己上传和练习过的 SQL 面试题。
2. 如果题目有明确出处，就保留出处。
3. 如果题目没有明确出处，就写：

   * 来源：用户上传题目
   * 原始出处：未提供
4. 不要为了补充出处去编造 LeetCode / 牛客题号。
5. 不要删除我原来写过的解答。
6. 如果旧文件里已经有“我的原始思路”或“我的解答”，必须原样保留，并迁移到新版模板的「我的解答」部分。
7. 如果旧文件没有我的解答，则写：

   * 本题暂未提供我的解答。
8. 所有标准答案统一使用 Hive SQL。
9. 标准答案必须是“包含 CTE 建表数据 + 解题 CTE + 最终 select”的完整可运行版本。
10. 不要再写 MySQL 标准答案。
11. 不要出现 MySQL 专属语法，例如：

    * date_sub(date, interval rn day)
    * date_add(date, interval n day)
    * cast(x as date) 的 MySQL 解释
12. Hive 日期函数要使用 Hive 写法，例如：

    * to_date(login_time)
    * date_sub(login_date, rn)
    * date_add(login_date, n)
    * datediff(date1, date2)
    * date_format(dt, 'yyyy-MM-dd')
13. 如果字段名是 `date`、`type` 等容易冲突的名字，可以在 SQL 中使用反引号，或者在 CTE 里改别名为更清晰的字段名，例如 login_time、login_date、event_date。
14. 所有 SQL 必须尽量写成适合面试复盘的分层 CTE，不要为了炫技压缩成一大段。

## 6. 标准答案：Hive CTE 完整版

这一节必须给出完整 Hive SQL。

要求：

1. 必须包含样例数据 CTE。
2. 必须包含解题过程 CTE。
3. 必须包含最终 select。
4. 必须能从头到尾独立运行。
5. 优先使用清晰的分层 CTE。
6. 每个 CTE 命名要有业务含义。
7. 不要使用 MySQL 专属语法。
8. 不要只给核心 select。
9. 不要省略建表数据。
10. 如果一题有多种写法，只保留最适合面试表达的一种主写法。

SQL 结构示例：

```sql
with data as (
    select stack(
        3,
        1, '2019-01-01 00:00:00',
        1, '2019-01-02 00:00:00',
        2, '2019-01-01 00:00:00'
    ) as (user_id, login_time)
),
base as (
    select
        user_id,
        to_date(login_time) as login_date
    from data
),
...
select
    ...
from ...
;
```

如果使用 `union all` 构造 CTE 数据也可以，但必须保证是 Hive 可运行写法。
